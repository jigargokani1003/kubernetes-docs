Pods — The Atomic Unit

A Pod is the smallest deployable unit. It wraps one or more containers that share an IP address, network namespace, and storage volumes. Every container in K8s lives inside a Pod.

nginx-pod.yaml — Production-ready
apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    app: nginx
    version: "1.25"
spec:
  containers:
  - name: nginx
    image: nginx:1.25
    ports:
    - containerPort: 80
    resources:          # ← ALWAYS set this!
      requests:
        cpu: "100m"
        memory: "128Mi"
      limits:
        cpu: "500m"
        memory: "256Mi"
    livenessProbe:     # Is it alive?
      httpGet:
        path: /healthz
        port: 80
      initialDelaySeconds: 10
      periodSeconds: 5
    readinessProbe:    # Ready to receive traffic?
      httpGet:
        path: /ready
        port: 80
      initialDelaySeconds: 5
      periodSeconds: 3
🧠 Memory Trick

Pod = Hotel Room. One IP (room number), multiple guests (containers) sharing bathroom (network + volumes). Hotel staff (kubelet) checks you're alive every few seconds (liveness probe). Concierge only sends guests when room is ready (readiness probe).

⚠️
Never run pods without resources.requests and resources.limits — your pod can starve other pods of CPU/memory and bring down the node!

deployment.yaml — Full production spec
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deploy
  annotations:
    kubernetes.io/change-cause: "Updated to nginx 1.25"   # shows in rollout history
spec:
  replicas: 3
  revisionHistoryLimit: 10    # keep 10 old ReplicaSets for rollback
  selector:
    matchLabels:
      app: nginx
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # +1 extra pod during update
      maxUnavailable: 0    # 0 pods down during update = zero downtime
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80
🧠 Memory Trick

Deployment = Git for Pods. set image = commit, rollout history = git log, rollout undo = git revert. You always have a safety net!

bash — Complete debug workflow
# Step 1: What's broken?
kubectl get pods -A | grep -Ev 'Running|Completed'

# Step 2: WHY is it broken? (Events section is gold!)
kubectl describe pod <pod-name>

# Step 3: What did the app say?
kubectl logs <pod-name> --previous 2>&1 | tail -50

# Step 4: Can I get inside?
kubectl exec -it <pod-name> -- /bin/sh

# Step 5: Is DNS working?
kubectl run dns-test --image=busybox:1.28 --rm -it --restart=Never -- nslookup kubernetes

# Step 6: Is service routing working?
kubectl get endpoints <service-name>    # Should NOT be empty!
kubectl run curl-test --image=curlimages/curl --rm -it --restart=Never -- curl http://<svc>:80

# Step 7: Resource pressure?
kubectl top nodes
kubectl describe node worker1 | grep -A5 "Allocated resources"

# Nuclear option: wipe and redeploy
kubectl rollout restart deployment/<name>

Topic 1Setup

KIND Cluster Setup

Install Docker, kubectl, KIND — and spin up a multi-node cluster locally in under 5 minutes.

📦 Install Docker + kubectl + KIND
bash
# 1. Install Docker
sudo apt-get update && sudo apt-get install -y docker.io
sudo systemctl enable --now docker && sudo usermod -aG docker $USER

# 2. Install kubectl
curl -LO "https://dl.k8s.io/release/$(curl -sL https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
chmod +x kubectl && sudo mv kubectl /usr/local/bin/

# 3. Install KIND
curl -Lo ./kind https://kind.sigs.k8s.io/dl/v0.22.0/kind-linux-amd64
chmod +x kind && sudo mv kind /usr/local/bin/

# 4. Create multi-node cluster
cat <<EOF > kind-config.yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane
  - role: worker
  - role: worker
EOF
kind create cluster --name my-cluster --config kind-config.yaml
kubectl get nodes
CommandPurposeExample
kind create clusterCreate a clusterkind create cluster --name dev --config kind.yaml
kind get clustersList clusterskind get clusters
kind delete clusterDestroy clusterkind delete cluster --name dev
kind load docker-imagePush local image to clusterkind load docker-image myapp:v1 --name dev
kind export kubeconfigGet kubeconfig for clusterkind export kubeconfig --name dev
🧠 Memory Trick

D-K-K = Docker, Kubectl, KIND — install in this order. Like building a house: foundation (Docker) → tools (kubectl) → blueprint (KIND cluster).

⚠️
Common Mistake: Forgetting sudo usermod -aG docker $USER then log out + log back in. Without this, KIND fails with permission denied.

Topic 4Workload

ReplicaSets — Self-Healing

A ReplicaSet ensures N pod replicas are always running. Kill a pod — it immediately creates a replacement. The supervisor that never sleeps.

CommandPurposeExample
kubectl get rsList ReplicaSetskubectl get rs -o wide
kubectl describe rsEvents + owned podskubectl describe rs nginx-rs
kubectl scale rsChange replica countkubectl scale rs nginx-rs --replicas=5
kubectl delete rsRemove RS + all podskubectl delete rs nginx-rs --cascade=foreground
🧠 Memory Trick

ReplicaSet = Security guard doing headcount every few seconds. Too few? Spawn more. Too many? Remove extras. Always hits target.

💡
Best practice: Don't use ReplicaSets directly — use Deployments which wrap ReplicaSets and add rolling updates + rollback.

Topic 7Config

Namespaces — Virtual Clusters

Namespaces divide one cluster into isolated virtual clusters. Same hardware, different teams, different environments — without paying for multiple clusters.

bash
kubectl get namespaces
kubectl create namespace dev
kubectl create namespace production
kubectl apply -f pod.yaml -n dev
kubectl get pods -n dev
kubectl get pods -A                     # all namespaces
kubectl config set-context --current --namespace=dev   # set default
# Cross-namespace DNS: service.namespace.svc.cluster.local
# e.g. db.production.svc.cluster.local
🧠 Memory Trick

Namespace = Office floor. Same building (cluster), different floors (namespaces). Talk to someone on another floor using full address: service.namespace.svc.cluster.local

⚠️
Danger: kubectl delete namespace production deletes ALL resources inside. Every pod, service, secret, config — gone. No confirmation. Be careful!

Topic 8Config

ConfigMaps

Decouple configuration from container images. Same Docker image, different behavior in dev vs prod — just swap the ConfigMap.

configmap.yaml + usage
# Create ConfigMap
kubectl create configmap app-config \
  --from-literal=LOG_LEVEL=debug \
  --from-literal=DB_URL=postgres://db:5432/app \
  --dry-run=client -o yaml > cm.yaml

# Use as env vars in pod
envFrom:
- configMapRef:
    name: app-config

# Use as mounted file
volumes:
- name: config-vol
  configMap:
    name: app-config
🧠 Memory Trick

ConfigMap = .env file for K8s. Don't hardcode config in Docker images. Externalize it so you can change behavior without rebuilding.

⚠️
Never put secrets here! ConfigMaps are stored plaintext in etcd. Use Secrets for passwords, tokens, and private keys.

Topic 9Security

Secrets

Store sensitive data (passwords, tokens, TLS certs) separately from code. Base64-encoded at rest, RBAC-protected access.

bash — Secrets workflow
kubectl create secret generic db-secret \
  --from-literal=username=admin \
  --from-literal=password=Sup3rS3cr3t!

kubectl get secret db-secret -o yaml
kubectl get secret db-secret -o jsonpath='{.data.password}' | base64 -d

# Use in pod
env:
- name: DB_PASSWORD
  valueFrom:
    secretKeyRef:
      name: db-secret
      key: password
🧠 Memory Trick

ConfigMap = plain envelope. Secret = sealed envelope. Both carry messages, but Secrets are sealed. Base64 is NOT encryption — it's just encoding. Enable etcd encryption-at-rest in prod!

Topic 10Storage

Volumes & Persistent Volumes

Container filesystem is ephemeral. PV (cluster storage) + PVC (your claim to that storage) = data that survives pod restarts. Like a network drive attached to your container.

pv-pvc-pod.yaml
# PV — the actual storage
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-local
spec:
  capacity:
    storage: 1Gi
  accessModes: [ReadWriteOnce]
  hostPath:
    path: /data/pv1
---
# PVC — your reservation
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 500Mi
🧠 Memory Trick

PV = Parking lot. PVC = Parking ticket. Pod = Your car. You claim a spot (PVC), the lot gives a real space (PV). Your car doesn't care which exact spot — just that it's parked.

Topic 11Networking

Ingress

One external IP, many backends. Route api.myapp.com → backend service, app.myapp.com → frontend — without creating a LoadBalancer per service.

ingress.yaml
# Install Nginx Ingress Controller for KIND
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/main/deploy/static/provider/kind/deploy.yaml
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - host: myapp.local
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 80
🧠 Memory Trick

Ingress = Hotel concierge. One front desk (Ingress controller), many rooms (services). You tell the concierge your destination (/api, /app) and they route you — no direct room number needed.

Topic 12Scale

Autoscaling — HPA

HPA watches CPU/memory. Too high → add pods. Too low → remove pods. Sleep through Black Friday traffic spikes.

hpa.yaml
# Quick imperative
kubectl autoscale deployment nginx --cpu-percent=70 --min=2 --max=10

# Declarative HPA
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: nginx-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: nginx
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
CommandPurposeExample
kubectl get hpaWatch scaling eventskubectl get hpa --watch
kubectl autoscaleQuick HPA creationkubectl autoscale deploy nginx --cpu-percent=70 --min=2 --max=10
kubectl top podsCurrent CPU/memorykubectl top pods --sort-by=cpu
🧠 Memory Trick

HPA = Elastic waistband. Sets min (can't go skinnier) and max (won't explode). Expands when you eat more (load), contracts when you don't. Always comfortable.

⚠️
HPA needs resources.requests.cpu set on containers. Without it, HPA can't calculate utilization percentage and stays inactive.
