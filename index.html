Kubernetes from Scratch

Using KIND Multi-Node Cluster

A hands-on guide to mastering Kubernetes one command at a time. Every topic has a concept, real-world example, command table, architecture diagram, and a memory trick so you never forget.

🎯 What is Kubernetes?

Kubernetes (K8s) is an open-source container orchestration system. It automates deployment, scaling, and management of containerized apps across a cluster of machines.

Think of it as an operating system for your data center — it schedules containers, heals failures, and exposes services automatically.

🐳 What is KIND?

Kubernetes IN Docker runs K8s nodes as Docker containers on your laptop. Perfect for learning without cloud costs.

  • Zero cost — runs locally
  • Multi-node clusters in seconds
  • Mirrors production behavior
  • Fast teardown and recreate

Topic 1Setup

KIND Cluster Setup

KIND (Kubernetes IN Docker) lets you run K8s nodes as Docker containers. Install 3 tools, write one config file, and you have a production-like multi-node cluster in under 5 minutes.

Why KIND?

Cloud clusters cost money. Minikube is single-node. KIND gives you a real multi-node cluster on your laptop — same behavior as production, free, and disposable. Perfect for practicing scheduling, node failures, and networking.

📦 Step 1 — Install Docker
bash – Linux (Ubuntu)
# Install Docker Engine
sudo apt-get update
sudo apt-get install -y docker.io
sudo systemctl enable --now docker
sudo usermod -aG docker $USER   # no sudo needed after re-login
docker --version                # Docker version 24.x.x
🔧 Step 2 — Install kubectl
bash
curl -LO "https://dl.k8s.io/release/$(curl -sL https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
chmod +x kubectl && sudo mv kubectl /usr/local/bin/
kubectl version --client   # Client Version: v1.29.x
⎈ Step 3 — Install KIND
bash
curl -Lo ./kind https://kind.sigs.k8s.io/dl/v0.22.0/kind-linux-amd64
chmod +x kind && sudo mv kind /usr/local/bin/
kind --version   # kind v0.22.0
🏗️ Step 4 — Create Multi-Node Cluster
kind-config.yaml
# kind-config.yaml — 1 control-plane + 2 workers
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane
  - role: worker
  - role: worker
bash
kind create cluster --name my-cluster --config kind-config.yaml
kubectl cluster-info --context kind-my-cluster
kubectl get nodes   # Should show 3 nodes (Ready)

🧠 Memory Trick

D-K-K = Docker first, then Kubectl, then KIND. Like building a house: foundation → tools → blueprint. You can't schedule what you can't run!

⚠️
Common Mistake: Forgetting sudo usermod -aG docker $USER → all KIND commands fail with permission denied. Always log out and back in after this step.
Topic 2CLI

Real-World Example

Topic 3Workload

Pods

A Pod is the smallest deployable unit in Kubernetes. It wraps one (or more) containers that share network and storage. Every container in K8s runs inside a Pod.

Why Pods?

Docker runs containers. K8s runs Pods. Pods add a shared network namespace (one IP per pod), shared volumes, and lifecycle management. A sidecar container (logging, proxy) can live next to your app in the same Pod.

Real-World Example

An Nginx web server pod + a Filebeat sidecar container sharing /var/log/nginx via a shared volume. One pod IP, two processes working together.

nginx-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.25
    ports:
    - containerPort: 80
    resources:
      requests:
        memory: "64Mi"
        cpu: "250m"
      limits:
        memory: "128Mi"
        cpu: "500m"
bash
kubectl apply -f nginx-pod.yaml
kubectl get pods -o wide
kubectl describe pod nginx        # events, node, IP
kubectl exec -it nginx -- bash    # shell into pod
kubectl port-forward pod/nginx 8080:80  # local testing
kubectl delete pod nginx

Topic 4Workload

ReplicaSets

A ReplicaSet ensures a specified number of identical pod replicas are running at all times. If a pod dies, it creates a new one. If too many exist, it deletes extras.

Why ReplicaSets?

Single pods have no self-healing. A ReplicaSet is a supervisor — it watches pods via label selectors and reconciles actual vs desired count, making your app resilient to node and pod failures.

replicaset.yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx-rs
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80
bash
kubectl apply -f replicaset.yaml
kubectl get rs                      # Check ReplicaSet
kubectl get pods --show-labels      # See label selectors
kubectl scale rs nginx-rs --replicas=5  # Scale up
kubectl delete pod nginx-rs-xxxxx  # RS recreates it!
kubectl delete rs nginx-rs         # Deletes RS + all pods

kubectl describe rsEvents & pod statuskubectl describe rs nginx-rs

ReplicaSet = Security Guard for pods. It checks the headcount every few seconds. Too few? Spawn more. Too many? Send some home. Always hits the target number.

💡
Best Practice: Don't use ReplicaSets directly — use Deployments which wrap ReplicaSets and add rolling update + rollback capabilities.
Topic 5Workload

Deployments

A Deployment is the standard way to run stateless apps in K8s. It manages ReplicaSets to provide rolling updates, rollback, and self-healing — all declaratively.

Real-World Example

You have myapp:v1 running with 5 replicas. Update to v2 — K8s creates a new ReplicaSet, rolls pods over one by one, zero downtime. Something breaks? One command rolls back to v1.

deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deploy
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80
bash – Deploy, update, rollback
kubectl apply -f deployment.yaml
kubectl get deploy
kubectl rollout status deployment/nginx-deploy

# Rolling update
kubectl set image deployment/nginx-deploy nginx=nginx:1.26
kubectl rollout history deployment/nginx-deploy

# Rollback
kubectl rollout undo deployment/nginx-deploy
kubectl rollout undo deployment/nginx-deploy --to-revision=1

# Scale
kubectl scale deployment nginx-deploy --replicas=5

Topic 6Networking

Services

Pods are ephemeral and get new IPs on restart. A Service provides a stable virtual IP + DNS name that load-balances traffic to matching pods via label selectors.

Three Service Types

ClusterIP

Internal only. Pod-to-pod communication inside the cluster. Default type.

NodePort

Exposes on each node's IP at a static port (30000–32767). Accessible externally.

LoadBalancer

Cloud LB provisioned automatically (AWS ELB, GCP LB). External IP assigned.

ExternalName

Maps service to external DNS name. No proxy, just CNAME.

service.yaml – All three types
---
# ClusterIP (default) — internal only
apiVersion: v1
kind: Service
metadata:
  name: nginx-clusterip
spec:
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
---
# NodePort — external access
apiVersion: v1
kind: Service
metadata:
  name: nginx-nodeport
spec:
  type: NodePort
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
    nodePort: 30080
---
# LoadBalancer — cloud / metallb
apiVersion: v1
kind: Service
metadata:
  name: nginx-lb
spec:
  type: LoadBalancer
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
bash
kubectl apply -f service.yaml
kubectl get svc
kubectl describe svc nginx-clusterip
# Test NodePort in KIND
kubectl get nodes -o wide  # get node IP
curl http://NODE_IP:30080

kubectl describe svcEndpoints & selectorskubectl describe svc nginx
kubectl exposeCreate service from resourcekubectl expose deploy nginx --port=80 --type=NodePort
kubectl delete svcRemove servicekubectl delete svc nginx

Topic 7Organization

Namespaces

Namespaces divide a single K8s cluster into virtual clusters. Resources in different namespaces are isolated but can communicate using full DNS names.

Why Namespaces?

One cluster, multiple teams. Dev, staging, prod — or team-frontend, team-backend. Each namespace has its own resource quotas, RBAC rules, and service discovery scope. Saves money vs multiple clusters.

bash
kubectl get namespaces              # List all NS
kubectl create namespace dev        # Create dev NS
kubectl create namespace production

# Deploy into a namespace
kubectl apply -f pod.yaml -n dev
kubectl get pods -n dev
kubectl get pods -A                 # All namespaces

# Cross-namespace DNS
# service.namespace.svc.cluster.local
# e.g.: nginx.dev.svc.cluster.local

# Set default namespace for session
kubectl config set-context --current --namespace=dev
namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: dev
  labels:
    env: development
CommandPurposeExample
kubectl get nsList namespaceskubectl get ns
kubectl create nsCreate namespacekubectl create ns staging
kubectl -n NSTarget a namespacekubectl get pods -n production
kubectl delete nsDelete NS + all resourceskubectl delete ns dev
🧠 Memory Trick

Namespace = Office Floor. Same building (cluster), different floors (namespaces). You can call someone on another floor using their full address (service.namespace.svc.cluster.local), not just their name.

⚠️
Common Mistake: kubectl delete namespace production deletes EVERYTHING inside — all pods, services, configs, secrets. Always double-check!
Topic 8Config

ConfigMaps

ConfigMaps store non-sensitive configuration as key-value pairs, separate from the container image. Inject them into pods as environment variables or mounted files.

Real-World Example

Same Docker image deployed to dev and prod. ConfigMap holds DATABASE_URL, LOG_LEVEL, API_ENDPOINT. Change config without rebuilding the image — just update the ConfigMap and restart pods.

configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  LOG_LEVEL: "debug"
  DATABASE_URL: "postgres://db:5432/myapp"
  config.ini: |
    [app]
    port=8080
    debug=true
pod-with-configmap.yaml
spec:
  containers:
  - name: myapp
    image: myapp:v1
    # Method 1: Environment variables
    envFrom:
    - configMapRef:
        name: app-config
    # Method 2: Volume mount (for files)
    volumeMounts:
    - name: config-vol
      mountPath: /etc/config
  volumes:
  - name: config-vol
    configMap:
      name: app-config
bash
kubectl apply -f configmap.yaml
kubectl get configmap
kubectl describe configmap app-config
kubectl create configmap env-config --from-literal=ENV=prod
kubectl create configmap file-config --from-file=app.conf
🧠 Memory Trick

ConfigMap = .env file but for Kubernetes. You wouldn't hardcode your DB URL in your app code — don't hardcode it in your Docker image either. Externalize it!

⚠️
Common Mistake: ConfigMaps are NOT for passwords or API keys — those go in Secrets. ConfigMaps are stored in etcd unencrypted, visible to anyone with cluster access.
Topic 9Security

Secrets

Secrets store sensitive data (passwords, tokens, TLS certs) in base64-encoded form. K8s provides access controls and encryption-at-rest for Secrets, unlike ConfigMaps.

Real-World Example

Your app connects to a database. Store DB_PASSWORD and API_TOKEN in a Secret. Mount it as env vars or a file. No passwords in code, no passwords in Docker images, no passwords in Git history.

bash – Create secrets
# From literals (values auto base64-encoded)
kubectl create secret generic db-secret \
  --from-literal=username=admin \
  --from-literal=password=Sup3rS3cr3t!

# From file
kubectl create secret generic tls-secret \
  --from-file=tls.crt=server.crt \
  --from-file=tls.key=server.key

# View (base64 encoded)
kubectl get secret db-secret -o yaml
# Decode
kubectl get secret db-secret -o jsonpath='{.data.password}' | base64 -d
pod-with-secret.yaml
spec:
  containers:
  - name: app
    image: myapp:v1
    env:
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-secret
          key: password
    volumeMounts:
    - name: secret-vol
      mountPath: /run/secrets
      readOnly: true
  volumes:
  - name: secret-vol
    secret:
      secretName: db-secret
🧠 Memory Trick

ConfigMap = Plain envelope. Secret = Sealed envelope. Both carry messages (config). But Secrets are sealed (base64 + RBAC + optional encryption-at-rest). Don't put bank details in a plain envelope!

⚠️
Common Mistake: Base64 is NOT encryption! Anyone with cluster access can decode it. Enable etcd encryption-at-rest and use RBAC to restrict Secret access in production.
Topic 10Storage

Volumes & Persistent Volumes

Kubernetes volumes give containers a place to store data that outlives a container restart. PersistentVolumes (PV) and PersistentVolumeClaims (PVC) provide durable cluster-level storage.

Why Persistent Volumes?

Container filesystem is ephemeral — restart = data gone. PV is a cluster resource (like a USB drive). A PVC is a request for storage. Pods use PVCs, not PVs directly — decoupling workload from infrastructure.

pv-pvc.yaml
---
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-local
spec:
  capacity:
    storage: 1Gi
  accessModes:
  - ReadWriteOnce
  hostPath:
    path: /data/pv1    # KIND-friendly (hostPath)
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: pvc-app
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 500Mi
---
# Use PVC in a Pod
apiVersion: v1
kind: Pod
metadata:
  name: db-pod
spec:
  containers:
  - name: postgres
    image: postgres:15
    volumeMounts:
    - mountPath: /var/lib/postgresql/data
      name: db-storage
  volumes:
  - name: db-storage
    persistentVolumeClaim:
      claimName: pvc-app
bash
kubectl get pv       # Check PV status (Available/Bound)
kubectl get pvc      # Check PVC binding
kubectl describe pvc pvc-app
graph LR P["Pod"] -->|mounts| PVC["PVC\n(request: 500Mi)"] PVC -->|bound to| PV["PV\n(capacity: 1Gi)"] PV -->|backed by| ST["Storage\n(hostPath/NFS/EBS/GCS)"] style P fill:#7c3aed,stroke:#a78bfa,color:#fff style PVC fill:#1d4ed8,stroke:#60a5fa,color:#fff style PV fill:#065f46,stroke:#34d399,color:#fff style ST fill:#78350f,stroke:#fbbf24,color:#fff
🧠 Memory Trick

PV = Parking Lot. PVC = Parking Ticket. Pod = Car. You claim a spot (PVC), the lot gives you a real space (PV). Your car (Pod) doesn't care which exact spot — just that it's parked somewhere.

⚠️
Common Mistake: PVC stays Pending when no PV matches the access mode or storage size. Check kubectl describe pvc → Events section for exact reason.
Topic 11Networking

Ingress

Ingress manages external HTTP/HTTPS access to services using host/path-based routing rules. One Ingress replaces many LoadBalancer services — one external IP, many backends.

Real-World Example

api.myapp.com/users → backend service. api.myapp.com/orders → orders service. app.myapp.com → frontend service. All via ONE Nginx Ingress Controller and ONE external IP.

bash – Install Ingress Controller (KIND)
# Install NGINX Ingress Controller for KIND
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/main/deploy/static/provider/kind/deploy.yaml
kubectl wait --namespace ingress-nginx \
  --for=condition=ready pod \
  --selector=app.kubernetes.io/component=controller \
  --timeout=90s
ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - host: myapp.local
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 80
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-service
            port:
              number: 80

Topic 12Scale

Autoscaling

K8s has three autoscalers: HPA scales pod replicas based on CPU/memory, VPA adjusts pod resource requests, and Cluster Autoscaler adds/removes nodes. HPA is the most common.

Real-World Example

Black Friday traffic spikes 10x. HPA detects CPU > 70%, scales your Deployment from 3 → 20 pods automatically. Traffic drops? Scales back to 3. You sleep through the night.

bash – Enable metrics server (KIND)
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
# For KIND, patch to allow insecure TLS:
kubectl patch deployment metrics-server -n kube-system \
  --type='json' -p='[{"op":"add","path":"/spec/template/spec/containers/0/args/-","value":"--kubelet-insecure-tls"}]'
hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: nginx-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: nginx-deploy
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
bash
kubectl apply -f hpa.yaml
kubectl get hpa                          # Watch scaling events
kubectl top pods                         # CPU/Memory usage
kubectl top nodes
# Quick imperative HPA:
kubectl autoscale deployment nginx-deploy --cpu-percent=70 --min=2 --max=10
🧠 Memory Trick

HPA = Elastic waistband on your trousers. Sets minimum (can't go below), maximum (won't burst), and expands/contracts based on how much you eat (load). Comfortable in all situations!

⚠️
Common Mistake: HPA won't work without resources.requests.cpu set on the container. It needs a baseline to calculate percentage utilization.
Topic 13Package Manager

Helm Basics

Helm is the package manager for Kubernetes. Charts are templated YAML bundles. Instead of managing 20 raw YAML files, you install, upgrade, and rollback apps with one command.

Real-World Example

Install Prometheus + Grafana monitoring stack in one command: helm install monitoring prometheus-community/kube-prometheus-stack. Customise with values.yaml. Upgrade when new chart version releases. No manual YAML editing of 100+ files.

bash – Install Helm
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash
helm version
bash – Common Helm operations
# Add a chart repo
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update

# Search for charts
helm search repo nginx
helm search hub wordpress

# Install a chart
helm install my-nginx bitnami/nginx
helm install my-nginx bitnami/nginx --namespace prod --create-namespace

# Customize with values
helm install my-app ./mychart -f values-prod.yaml
helm install my-app ./mychart --set replicaCount=3

# Lifecycle management
helm list                          # Installed releases
helm status my-nginx               # Release status
helm upgrade my-nginx bitnami/nginx --set replicaCount=3
helm rollback my-nginx 1           # Rollback to revision 1
helm uninstall my-nginx            # Remove release

# Inspect before install
helm show values bitnami/nginx > values.yaml
helm template my-app ./mychart     # Render templates locally

🧠 Memory Trick

Helm = apt/brew for K8s. Just like apt install nginx installs Nginx on Linux, helm install deploys a full K8s app. Charts = packages. Values = config flags. Releases = installed apps.

Topic 14Operations

Debugging Kubernetes

When things break, follow a systematic top-down approach: Deployment → ReplicaSet → Pod → Container → Logs → Events. K8s always tells you what's wrong — you just need to ask the right questions.

🛠️ Debugging Toolkit
bash – Complete debugging workflow
# 1. Check overall health
kubectl get nodes
kubectl get pods -A | grep -v Running | grep -v Completed

# 2. Describe failing pod (events are gold!)
kubectl describe pod POD_NAME

# 3. Application logs
kubectl logs POD_NAME
kubectl logs POD_NAME --previous          # Crashed container logs
kubectl logs POD_NAME -c CONTAINER_NAME  # Multi-container pod
kubectl logs POD_NAME -f --tail=100       # Stream last 100 lines

# 4. Shell into container
kubectl exec -it POD_NAME -- bash
kubectl exec -it POD_NAME -- sh          # If no bash

# 5. Run a debug sidecar
kubectl debug -it POD_NAME --image=busybox --target=mycontainer

# 6. Check service endpoints
kubectl get endpoints SVC_NAME
kubectl describe svc SVC_NAME

# 7. Check events (most useful!)
kubectl get events --sort-by='.lastTimestamp' -n default
kubectl get events --field-selector type=Warning

# 8. Test DNS from inside cluster
kubectl run dns-test --image=busybox --rm -it --restart=Never -- nslookup kubernetes

# 9. Resource usage
kubectl top pods
kubectl top nodes

# 10. Check resource quotas
kubectl describe resourcequota -n NAMESPACE
